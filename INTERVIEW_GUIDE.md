**技术细节**：
- pgvector 余弦相似度检索

## 🗣️ 常见面试问题及回答

### Q1: 为什么要从微服务改回单体？

**回答**：
> "原来的架构有几个问题：第一，7 个微服务共享一个数据库，违背了微服务的数据隔离原则；第二，RocketMQ 只用于 2 个简单场景，过度设计；第三，ShardingSphere 在单数据库上分表，没有实际性能提升。对于一个演示项目，这些复杂度带来的成本远大于收益。所以我重构为模块化单体，保留了清晰的模块边界，但去掉了不必要的分布式复杂度。这让启动时间提升了 6 倍，内存占用减少 75%。"

---

### Q2: RAG 和传统搜索有什么区别？

**回答**：
> "传统搜索是关键词匹配，比如搜'Redis'只能找到包含这个词的笔记。RAG 是语义搜索，它理解问题的含义。比如用户问'如何防止缓存击穿'，即使笔记里没有'击穿'这个词，只要内容相关（比如提到了'布隆过滤器'），也能检索到。技术上，我用 Embedding API 将文本转为向量，用 pgvector 的余弦相似度检索，然后用 LLM 基于检索到的上下文生成答案。这是检索增强生成，比纯 LLM 更准确，比传统搜索更智能。"

---

### Q3: 知识图谱是怎么构建的？

**回答**：
> "我用 LLM 做实体和关系提取。首先设计 Prompt，明确定义 4 种实体类型（人员、项目、流程、概念）和 4 种关系类型（负责、依赖、前置、相关），要求 LLM 返回结构化 JSON。然后解析 JSON，存储到 PostgreSQL 的 knowledge_entity 和 knowledge_relation 表。每个实体和关系都有置信度评分。前端用 SVG 渲染，实现了圆形布局算法。这展示了我对 Prompt Engineering 和结构化知识表示的理解。"

---

### Q4: 如果没有 AI API Key 怎么演示？

**回答**：
> "我实现了 Mock 模式。如果检测到没有 API Key，系统会自动切换到 Mock 模式：Embedding 用基于文本 hash 的确定性向量，LLM 返回检索到的笔记内容摘要。这样即使没有 API Key，整个流程也能跑通，方便开发和演示。这是我对开发体验的考虑。"

---

### Q5: 这个项目的技术难点是什么？

**回答**：
> "主要有三个难点：第一，pgvector 的向量检索优化，我用了 IVFFlat 索引，并且调整了 lists 参数；第二，异步向量化的设计，要保证不阻塞用户操作，同时处理失败重试；第三，知识图谱的 Prompt Engineering，要让 LLM 稳定输出结构化 JSON，我加了 JSON Schema 示例和错误处理。这些都是生产级系统需要考虑的细节。"

- 如果 API 失败，切换到 Mock 模式
- 如果时间不够，优先展示 RAG

### 4. 展示代码（可选）
```java
// 展示 RAG 核心代码
@Service
public class RagQueryService {
    public RagQueryResponse query(Long userId, RagQueryRequest request) {
        // 1. 向量检索
        List<SimilarNote> notes = vectorRepository.searchSimilarNotes(...);
        
        // 2. 构建上下文
        String context = buildContext(notes);
        
        // 3. LLM 生成答案
        String answer = generateAnswer(request.getQuery(), context);
        
        // 4. 返回结果 + 来源
        return new RagQueryResponse(answer, citations, ...);
    }
}
```
