# 项目讲稿

下面这三套话术不是让你死背，而是先背出一个稳定版本，再逐步改成你自己的表达。

## 90 秒自我介绍

大家好，我是软件工程专业的应届生，主攻方向是 Java 后端开发。最近一段时间我主要在做一个叫 `LifeOS` 的项目，这是一个面向个人知识管理的微服务系统，核心是把笔记、AI 整理、任务转化和行为统计串成一个闭环。

技术上我主要使用了 `Java 17`、`Spring Boot 3`、`Spring Cloud Gateway`、`Nacos`、`MyBatis-Plus`、`MySQL`、`Redis` 和 `RocketMQ`。在这个项目里，我重点关注了几个方向：一是基于 `JWT + Redis + Gateway` 的统一鉴权；二是用 `RocketMQ` 处理 AI 作业和行为事件的异步链路；三是结合业务场景做 Redis 缓存和登录限流。

我现在找的是 Java 后端开发岗位，希望进入团队后先把常规 Web 开发、数据库、缓存和接口协作这些基础工作做好，同时继续提升自己在微服务和分布式场景下的工程能力。

## 3-5 分钟项目介绍

我做的项目叫 `LifeOS`，它不是一个通用聊天工具，而是一个面向个人知识管理的系统。用户可以记录笔记、触发 AI 摘要和整理、从笔记里提取任务，任务完成之后再回写行为数据，最后在 Dashboard 里展示最近的任务、待复习笔记和行为趋势。

整个后端我做成了一个多模块微服务项目。前端统一访问网关，网关负责统一路由和登录态校验。后端按业务拆成了用户服务、笔记服务、任务服务、AI 服务和行为分析服务。公共 DTO、Feign 接口和消息常量放在 `lifeos-api`，JWT 工具和统一返回结构放在 `lifeos-common`。

这个项目里我最想讲的第一个点是统一鉴权。用户登录时，用户服务会校验账号密码，然后生成 JWT。同时我会把 token 写到 Redis 里。后续所有请求先经过 Gateway，网关先校验 JWT 的签名和过期时间，然后再去 Redis 校验这个 token 是否还是当前有效登录态。通过后会把 `X-User-Id` 之类的请求头传给下游服务。这样做的好处是，既保留了 JWT 的轻量表达能力，又能支持退出登录、强制失效和单端登录控制。

第二个点是异步链路。比如用户触发笔记摘要或者任务提取时，笔记服务不会同步等待 AI 返回，而是先把作业记录落库，再把任务发到 RocketMQ。AI 服务消费消息后完成处理，再通过 Feign 回调笔记服务更新作业状态和结果。这样可以避免 AI 耗时直接阻塞主请求，也更方便做失败重试和状态展示。

第三个点是 Redis 的实际使用。除了登录态，我还在任务服务里对任务列表做了缓存。查询任务列表时先查 Redis，未命中再查 MySQL，然后回填缓存；新增、更新、删除任务时主动删缓存。这个方案实现成本低，比较适合当前这种中小规模项目。另外登录接口还加了基于 Redis 的限流控制，按用户名限制每分钟最多尝试 5 次，防止暴力尝试。

数据库方面，项目里也做了一些面向查询场景的索引设计。比如笔记按用户、创建时间、置顶状态、复习状态建立组合索引，任务和 AI 作业也按用户和时间维度做了索引。我的理解是，校招项目里如果能把“索引为什么这么建”讲清楚，会比只会背 B+ 树更有说服力。

如果让我总结这个项目的价值，我会说它体现了我对企业后端几个常用问题的理解：统一鉴权、服务拆分、异步解耦、缓存使用和面向查询的表设计。

## 10 分钟技术深挖

### 为什么拆微服务，而不是先做单体

这个项目里用户、笔记、任务、AI 和行为统计的职责边界比较清晰，所以我选择按领域拆服务。这样做的主要好处有三个。第一是职责更清楚，比如 AI 服务只负责 AI 处理，行为服务只负责统计聚合。第二是便于异步解耦，像 AI 作业和行为事件都可以围绕单一职责服务展开。第三是后续扩展更自然，比如后台管理、更多 AI 能力、更多统计维度都能按服务演进。

当然微服务也会带来成本，比如部署复杂、服务治理、调用链追踪更麻烦，所以如果这是一个很小的业务，我也会考虑先做单体。但在这个项目里，我是把它当作一个面向校招展示的工程化项目来做，所以有意识地保留了网关、注册中心、消息队列这些企业常见组件。

### 为什么引入 RocketMQ

最典型的场景是 AI 作业。AI 摘要和整理这种操作耗时不稳定，如果用户请求同步等待，会导致接口很慢，体验不好，也不利于失败重试。所以我在笔记服务里采用了“先落作业表，再发 MQ”的方式。这样用户立刻能拿到作业 ID 和状态，AI 服务异步消费并更新结果。

第二个场景是行为事件。任务完成、笔记整理、从笔记提取任务这些动作，本身的主业务已经完成了，不应该再同步等待统计逻辑。用 MQ 后，主流程只负责发一个行为事件，行为服务自己异步消费并聚合 Dashboard。这样主流程更轻，统计逻辑也更独立。

### Redis 和数据库一致性怎么处理

当前项目里主要是旁路缓存模式。以任务列表为例，读的时候先查 Redis，没有就查数据库并回填；写的时候新增、更新、删除任务后删缓存。这个方案的优点是简单直接，很适合中小项目。它不是绝对强一致的，理论上仍然存在短暂脏读窗口，但对任务列表场景是可接受的。

如果后续这个系统的数据量更大或者一致性要求更高，我会考虑两种增强方案：一种是延迟双删，降低并发更新场景下的脏数据窗口；另一种是把缓存更新也做成事件驱动，让更新路径更可控。

### 网关和各服务分别负责什么

网关负责横切逻辑，主要是统一入口、路由转发、登录态校验和 Swagger 聚合。它不负责业务本身。业务服务只处理自己的领域逻辑，比如用户服务只管登录注册、笔记服务只管笔记和 AI 作业提交、任务服务只管任务、行为服务只做统计。

这样拆分的好处是，鉴权和路由只做一遍，业务服务更专注。如果以后要加限流、黑名单、统一日志、灰度规则，这些也更适合放在网关层统一处理。

## 常见追问短答

### 你这个登录为什么不是纯 JWT

因为纯 JWT 无法优雅支持退出登录和强制失效。这里加 Redis 主要是为了可控的登录态管理。

### 如果 Redis 挂了怎么办

短期内会影响登录态校验。我会根据业务重要性决定降级策略。当前这个项目里更偏向安全优先，宁可校验失败也不放过未确认请求。

### AI 服务为什么不直接写笔记库

因为那会破坏服务边界。AI 服务只负责处理，结果还是通过接口回到笔记服务更新，这样职责更清晰。

### 行为事件如何避免重复消费

事件带 `eventId`，行为服务落库时对重复事件做幂等兜底，捕获 `DuplicateKeyException` 后忽略。
